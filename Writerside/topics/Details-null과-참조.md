# Details) null과 참조, 그리고 원시 타입

[Q0) null 안정성](Q0-null-안정성.md)을 이해하려면 먼저 **`null`이 무엇인지**, **"null을 참조한다"는 게 어떤 의미인지**를 짚어야 합니다. 그런데 이 개념을 따라가다 보면 자연스럽게 한 가지 의문에 부딪힙니다. **`Int`나 `Boolean` 같은 원시 타입은 애초에 `null`이 될 수 없는 것 아닐까?** 이 문서는 그 의문을 끝까지 밀어붙여서, Kotlin의 null 안정성이 실제로 무엇 위에 서 있는지까지 도달합니다.

## null이란 무엇인가 {#what-is-null}

**`null`은 "값이 없음"을 나타내는 특별한 값**입니다. 숫자 `0`이나 빈 문자열 `""`과는 다릅니다. `0`은 "0이라는 값이 있는" 상태이고 `""`은 "길이가 0인 문자열이 있는" 상태지만, `null`은 **가리킬 대상 자체가 없는** 상태입니다.

## 참조(reference)란 무엇인가 {#what-is-reference}

`null`을 이해하려면 **참조(reference)** 개념이 필요합니다. 객체 타입의 변수는 객체를 변수 안에 직접 담지 않습니다. 대신 메모리 어딘가(힙)에 놓인 객체의 **주소(위치)를 가리키는 참조**를 담습니다.

변수를 **사물함 열쇠**라고 생각하면 쉽습니다. 열쇠 자체는 물건이 아니라, 물건이 든 사물함을 가리킬 뿐입니다.

```
name ──▶ [ "Kotlin" 객체 ]   // name은 문자열 객체를 가리킨다
addr ──▶ ✗ (null)            // addr은 아무것도 가리키지 않는다
```

## "null을 참조한다"는 것의 의미 {#dereferencing-null}

**`null` 참조**란 이렇게 **아무 객체도 가리키지 않는 참조**입니다. 열쇠는 손에 있는데 정작 열 사물함이 없는 상태죠.

그래서 **"null을 참조(역참조, dereference)한다"** 는 것은, 가리키는 대상이 없는 이 참조를 통해 객체의 멤버(프로퍼티·메서드)에 접근하려는 시도를 뜻합니다. `null.length`처럼요. 존재하지 않는 사물함을 열어 내용물을 꺼내려는 것과 같아서, 실행하면 오류가 납니다. 이 오류가 바로 `NullPointerException`(NPE)입니다.

뒤집어 보면, **NPE는 참조를 거치는 타입에서만 일어날 수 있는 사고**입니다. 열쇠라는 간접 단계가 있어야 "열쇠는 있는데 열 사물함이 없다"는 상황도 생기니까요.

## 그럼 원시 타입은 nullable하게 만들 수 없을까? {#can-primitives-be-nullable}

여기서 자연스러운 의문이 생깁니다. `Int`나 `Boolean` 같은 **원시(primitive) 타입**은 값을 변수에 직접 담습니다. 사물함에 넣고 열쇠를 주는 게 아니라, 작은 물건은 그냥 손에 쥐여 주는 셈이죠.

그렇다면 참조라는 간접 단계 자체가 없습니다. 참조가 없으면 "아무것도 가리키지 않는 참조"도 있을 수 없고, 그러면 `null`도 있을 수 없습니다. 손에 쥔 물건에는 "사물함이 비었다"는 상태가 존재하지 않으니까요.

**그럼 원시 타입은 애초에 nullable하게 만들 수 없는 것 아닐까요?**

### 그런데 Int?는 컴파일된다 {#int-nullable-compiles}

추론은 그럴듯하지만, 실제로 써 보면 아무 문제 없이 컴파일됩니다.

```kotlin
val nullableCount: Int? = null   // 컴파일된다
```

추론의 어딘가가 틀렸다는 뜻입니다. 어디일까요?

### 첫 번째 층 — Kotlin 소스에는 원시 타입이 없다 {#no-primitives-in-source}

Kotlin에는 소스 코드 수준에서 **원시 타입이 따로 없습니다.** `Int`, `Boolean`, `Double`은 전부 클래스이고, 메서드도 가집니다.

```kotlin
val n: Int = 5
n.toString()          // Int에 메서드가 있다
n.coerceAtLeast(0)    // 5
true.not()            // Boolean도 마찬가지
```

Java가 `int`(원시 타입)와 `Integer`(객체)를 나눠 두는 것과 달리, Kotlin은 이 구분을 겉으로 드러내지 않습니다. `Int`가 그냥 클래스라면 `Int?`라고 쓰는 데 아무 제약이 없습니다.

하지만 이건 겉모습일 뿐입니다. JVM에는 여전히 `int`가 있고, `int`를 담는 4바이트 자리에는 "값이 없음"을 표현할 여유 공간이 없습니다. 그래서 진짜 질문은 이겁니다. **JVM에서 `int`에 어떻게 `null`을 넣는가?**

### 두 번째 층 — 넣지 않는다, 참조로 바꾼다 {#boxing}

답은 **넣지 않는다**입니다. 컴파일러는 `Int?`를 만나면 그 값을 `int`가 아니라 박싱된 `java.lang.Integer`로 낮춥니다.

| Kotlin 선언 | JVM 실제 표현 | 저장 방식 |
|---|---|---|
| `Int` (non-null) | 원시 타입 `int` | **값을 변수에 직접** 담음 (참조 아님) |
| `Int?` (nullable) | 박싱된 `java.lang.Integer` | 힙 객체를 **참조**로 가리킴 |

즉 "원시 타입을 nullable하게 만들 수 있나?"라는 질문에 대한 컴파일러의 대답은 **"그럼 참조로 바꿔주지"** 입니다.

사물함 비유를 그대로 끌고 오면 이렇습니다. 작은 물건은 원래 손에 쥐여 주지만, `null`일 수 있어야 한다고 선언하면 굳이 사물함에 넣고 열쇠를 줍니다. 그래야 "열쇠는 있는데 사물함이 없음"이라는 상태를 표현할 수 있으니까요.

그러니 앞선 추론에서 틀린 곳은 **"`Int`는 원시 타입이다"** 라는 전제였습니다. `Int`가 원시 타입인지 아닌지는 고정된 사실이 아니라, `null`을 담을 수 있는지에 따라 달라집니다.

## 표현이 아니라 타입이 정한다 {#type-decides}

여기서 인과의 방향을 눈여겨볼 만합니다.

`Int`가 원시 타입이라서 `null`을 못 갖는 게 아닙니다. **`null`을 가져도 된다고 선언(`Int?`)했기 때문에 원시 타입이 아니게 된 것**입니다. 값이냐 참조냐가 규칙을 정하는 게 아니라, 규칙이 값이냐 참조냐를 정합니다.

```kotlin
var count: Int = 0
count = null    // 컴파일 에러 — Int로 선언했으니 null이 들어올 수 없다

var nullableCount: Int? = 0
nullableCount = null   // OK — Int?로 선언했으니 컴파일러가 참조로 표현해 준다
```

이 방향이 맞다는 것은 반대편 두 사례로 확인됩니다.

- **`String`은 참조 타입인데도 `null`이 막힙니다.** JVM 바이트코드 수준에서 `String` 자리에 `null`을 넣는 건 완전히 합법입니다. 표현상으로는 아무 문제가 없죠. 그런데도 `var name: String = null`은 컴파일 에러입니다. 막는 주체는 표현이 아니라 컴파일러입니다.
- **`Int`는 원시 타입이 없는 곳에서도 `null`이 막힙니다.** Kotlin/Native에는 JVM의 `int`/`Integer` 구분 자체가 없습니다. 그런데도 `Int`에는 여전히 `null`을 대입할 수 없습니다.

> **Java는 어땠나** — Java도 `int x = null;`은 컴파일 에러입니다. `null` 리터럴이 참조 타입에만 대입될 수 있다는 **타입 검사**에 걸리는 것이지, 메모리에 자리가 없어서가 아닙니다. 다만 Java의 보호는 딱 거기까지라 `String s = null;`은 통과하고, `Integer i = null; int x = i;`조차 오토언박싱으로 컴파일을 통과한 뒤 런타임에 NPE가 납니다. Kotlin이 새로 한 일은 `Int`를 막은 게 아니라, **같은 규칙을 `String`에까지 확장한 것**입니다.

두 경우 모두 런타임 표현은 아무 결정권이 없습니다. 결정하는 것은 오직 **선언한 타입이 `T`냐 `T?`냐**입니다. `T`와 `T?`는 서로 다른 타입이고, `null`은 `T?`에만 속하는 값이며, 컴파일러가 이 구분을 강제합니다. 박싱과 언박싱은 그 규칙을 각 플랫폼에서 지켜내기 위해 컴파일러가 알아서 고르는 뒷단의 구현일 뿐입니다.

이것이 [Q0) null 안정성](Q0-null-안정성.md)이 서 있는 토대입니다. Q0에서 만나게 될 `?`, 안전 호출(`?.`), Elvis(`?:`)는 전부 참조냐 값이냐와 무관하게 **타입에만** 걸립니다. 개발자가 상대하는 것은 메모리 표현이 아니라 타입 하나뿐입니다.

> **참고** — 박싱/언박싱은 **JVM 백엔드**의 이야기이고, Kotlin/Native나 Kotlin/JS는 값 표현 방식이 다릅니다. 하지만 "non-null 타입에는 `null`을 담을 수 없다"는 언어 차원의 규칙은 모든 플랫폼에서 동일합니다. 규칙이 표현에 앞선다는 사실이 여기서도 드러납니다.
