# Details) null과 참조, 그리고 원시 타입

[Q0) null 안정성](Q0-null-안정성.md)을 이해하려면 먼저 **`null`이 무엇인지**, **"null을 참조한다"는 게 어떤 의미인지**를 짚어야 합니다. 그리고 Kotlin에서는 `Int`·`Boolean` 같은 원시 타입도 이 그림에 어떻게 들어맞는지가 null 안정성과 직접 연결됩니다.

## null이란 무엇인가 {#what-is-null}

**`null`은 "값이 없음"을 나타내는 특별한 값**입니다. 숫자 `0`이나 빈 문자열 `""`과는 다릅니다. `0`은 "0이라는 값이 있는" 상태이고 `""`은 "길이가 0인 문자열이 있는" 상태지만, `null`은 **가리킬 대상 자체가 없는** 상태입니다.

## 참조(reference)란 무엇인가 {#what-is-reference}

`null`을 이해하려면 **참조(reference)** 개념이 필요합니다. 객체 타입의 변수는 객체를 변수 안에 직접 담지 않습니다. 대신 메모리 어딘가(힙)에 놓인 객체의 **주소(위치)를 가리키는 참조**를 담습니다.

변수를 **사물함 열쇠**라고 생각하면 쉽습니다. 열쇠 자체는 물건이 아니라, 물건이 든 사물함을 가리킬 뿐입니다.

```
name ──▶ [ "Kotlin" 객체 ]   // name은 문자열 객체를 가리킨다
addr ──▶ ✗ (null)            // addr은 아무것도 가리키지 않는다
```

## "null을 참조한다"는 것의 의미 {#dereferencing-null}

**`null` 참조**란 이렇게 **아무 객체도 가리키지 않는 참조**입니다. 열쇠는 손에 있는데 정작 열 사물함이 없는 상태죠.

그래서 **"null을 참조(역참조, dereference)한다"** 는 것은, 가리키는 대상이 없는 이 참조를 통해 객체의 멤버(프로퍼티·메서드)에 접근하려는 시도를 뜻합니다. `null.length`처럼요. 존재하지 않는 사물함을 열어 내용물을 꺼내려는 것과 같아서, 실행하면 오류가 납니다. 이 오류가 바로 `NullPointerException`(NPE)입니다.

## 그럼 Int, Boolean 같은 원시 타입은? {#primitive-types}

여기서 자연스러운 의문이 생깁니다. "객체 타입의 변수는 참조를 담는다"고 했는데, `Int`나 `Boolean` 같은 **원시(primitive) 타입**도 여기에 해당할까요? 답은 **바라보는 관점에 따라 다릅니다.**

### Kotlin 언어 관점 — 모두 객체다 {#language-view}

Kotlin에는 소스 코드 수준에서 **원시 타입이 따로 없습니다.** `Int`, `Boolean`, `Double`은 전부 클래스이고, 메서드도 가집니다.

```kotlin
val n: Int = 5
n.toString()          // Int에 메서드가 있다
n.coerceAtLeast(0)    // 5
true.not()            // Boolean도 마찬가지
```

즉 언어 모델상으로는 모든 것이 객체입니다. Java가 `int`(원시 타입)와 `Integer`(객체)를 나눠 두는 것과 달리, Kotlin은 겉보기에 이 구분이 없습니다.

### JVM 런타임 관점 — 값이냐 참조냐로 나뉜다 {#runtime-view}

하지만 컴파일러가 이를 JVM 바이트코드로 낮출 때는 성능을 위해 최적화가 들어갑니다. 이때 **nullable 여부**에 따라 표현이 갈립니다.

| Kotlin 선언 | JVM 실제 표현 | 저장 방식 |
|---|---|---|
| `Int` (non-null) | 원시 타입 `int` | **값을 변수에 직접** 담음 (참조 아님) |
| `Int?` (nullable) | 박싱된 `java.lang.Integer` | 힙 객체를 **참조**로 가리킴 |

정리하면, "변수는 객체의 참조를 담는다"는 설명은 엄밀히 말하면 **참조 타입**(그리고 박싱된 값)에만 정확히 들어맞습니다. non-null `Int`는 참조를 거치지 않고 값 자체를 바로 들고 있습니다. 사물함 비유로 치면, 열쇠(참조)를 주는 게 아니라 작은 물건은 그냥 손에 쥐여 주는 셈입니다.

## 이것이 null 안정성과 이어진다 {#link-to-null-safety}

바로 이 "값이냐 참조냐"의 차이가 [Q0) null 안정성](Q0-null-안정성.md)의 근본 원리와 맞닿습니다.

- **`Int` → 원시 타입 `int`** : 값을 직접 담는 자리에는 **`null`을 담을 공간이 없습니다.** 그래서 `Int`에는 `null`을 대입할 수 없습니다.
- **`Int?` → 박싱된 `Integer`(참조 타입)** : 컴파일러가 참조 타입으로 만들기 때문에, 그 **참조가 아무것도 가리키지 않는(null) 상태**가 가능합니다.

```kotlin
var count: Int = 0
count = null    // 컴파일 에러 — int에는 null 자리가 없다

var nullableCount: Int? = 0
nullableCount = null   // OK — Integer 참조가 null을 가리킨다
```

`var name: String = null`이 에러인 이유(참조 타입이지만 non-null 선언이라 컴파일러가 막음)와, `Int`가 애초에 `null`을 가질 수 없는 이유(원시 타입이라 null 자리 자체가 없음)는 결국 **"참조 vs 값"이라는 같은 뿌리**에서 나옵니다.

> **참고** — 이 박싱/언박싱 최적화는 **JVM 백엔드** 이야기입니다. Kotlin/Native나 Kotlin/JS는 값 표현 방식이 다릅니다. 다만 "non-null 원시값은 null을 가질 수 없다"는 언어 차원의 규칙은 모든 플랫폼에서 동일합니다.
