# E11) Scope Functions와 확장 함수

스코프 함수(scope functions)는 **어떤 객체를 받아, 그 객체를 컨텍스트로 하는 람다를 실행한 뒤, 객체 자신이나 람다 결과를 돌려주는 표준 라이브러리 함수**입니다. 확장 함수(extension function)는 **원본 클래스를 수정하지 않고 외부에서 새 함수를 추가한 것처럼 호출하게 해 주는 문법**입니다. 둘은 별개의 기능이지만, 스코프 함수가 전부 확장 함수로 구현되어 있고 동일한 "수신 객체(receiver)" 개념 위에 서 있기 때문에 함께 이해하는 것이 정확합니다.

이 토픽의 핵심은 다섯 스코프 함수가 **수신 객체를 무엇으로 노출하는가(`this` vs `it`)**와 **무엇을 반환하는가(객체 자신 vs 람다 결과)** 두 축으로 갈린다는 점, 그리고 확장 함수가 **인스턴스 메서드가 아니라 정적 디스패치되는 함수**라는 점입니다.

## 다섯 스코프 함수를 가르는 두 축 {#scope-axes}

다섯 함수의 차이는 외워야 하는 목록이 아니라, 두 가지 설계 선택의 조합입니다.

1. **람다 안에서 대상 객체를 무엇으로 가리키는가**
   - **수신 객체(`this`)로 노출**: `run`, `apply`, `with`. 람다 안에서 객체의 멤버를 `this` 생략으로 바로 부릅니다.
   - **인자(`it`)로 노출**: `let`, `also`. 람다 파라미터로 받으므로 이름을 바꿔 줄 수 있습니다.
2. **무엇을 반환하는가**
   - **람다의 결과를 반환**: `let`, `run`, `with`. 변환·계산 결과를 다음으로 넘길 때 씁니다.
   - **수신 객체 자신을 반환**: `apply`, `also`. 객체를 그대로 흘려보내며 부수 작업만 끼워 넣을 때 씁니다.

| 함수 | 객체 노출 | 반환값 | 확장 함수 여부 |
|------|-----------|--------|----------------|
| `let` | `it` | 람다 결과 | 예 (`T.let`) |
| `run` | `this` | 람다 결과 | 예 (`T.run`) |
| `with` | `this` | 람다 결과 | 아니오 (인자로 받음) |
| `apply` | `this` | 수신 객체 자신 | 예 (`T.apply`) |
| `also` | `it` | 수신 객체 자신 | 예 (`T.also`) |

이 표 한 장이 다섯 함수의 전부입니다. 아래에서 각각의 실제 시그니처와 쓰임을 확인합니다.

## let: it · 람다 결과 반환 {#fn-let}

`let`은 객체를 `it`으로 받고, **람다의 결과를 반환**합니다. 시그니처는 다음과 같습니다.

```kotlin
inline fun <T, R> T.let(block: (T) -> R): R = block(this)
```

가장 흔한 두 용도가 있습니다.

**1) nullable 객체에 대한 안전 호출.** `?.let { }`은 객체가 null이 아닐 때만 블록을 실행합니다.

```kotlin
val name: String? = user.name
name?.let {
    // it: String (안전 호출 ?.가 null을 걸러내므로 it은 non-null)
    println("길이: ${it.length}")
}
```

`?.let`이 동작하는 이유는 스마트캐스트가 아니라 **안전 호출 연산자 `?.`** 입니다. `name`이 null이면 `?.`가 `let` 호출 자체를 건너뛰고 전체 식이 null이 됩니다.

**2) 임시 변수 없이 값을 변환해 넘기기.**

```kotlin
val firstUpper: Char? = text.firstOrNull()?.let { it.uppercaseChar() }
```

## run: this · 람다 결과 반환 {#fn-run}

`run`은 객체를 `this`로 받고, **람다의 결과를 반환**합니다. `let`과 반환 정책은 같지만 객체를 `this`로 노출한다는 점이 다릅니다.

```kotlin
inline fun <T, R> T.run(block: T.() -> R): R = block()
```

`block: T.() -> R`은 **수신 객체가 있는 람다(lambda with receiver)** 타입입니다. 그래서 블록 안에서 객체의 멤버를 `this` 없이 바로 부릅니다.

```kotlin
val area: Int = rect.run {
    width * height   // this.width * this.height
}
```

`run`에는 **확장 함수가 아닌 형태**도 있습니다. 수신 객체 없이 블록을 실행하고 결과를 반환하는 버전으로, 여러 문장을 묶어 하나의 값으로 만들 때 씁니다.

```kotlin
val config = run {
    val raw = load()
    parse(raw)        // 이 결과가 config가 된다
}
```

## with: this · 비확장 · 람다 결과 반환 {#fn-with}

`with`는 객체를 **인자로 받는** 함수입니다. 확장 함수가 아니라는 점만 빼면 `run`과 동작이 같습니다. 객체를 `this`로 노출하고 람다 결과를 반환합니다.

```kotlin
inline fun <T, R> with(receiver: T, block: T.() -> R): R = receiver.block()
```

```kotlin
val description = with(user) {
    "$name ($age세)"   // this.name, this.age
}
```

`run`과의 선택 기준은 단순합니다. 대상이 null일 수 있으면 `?.run`이 자연스럽고, null이 아닌 객체를 그룹으로 다룰 때는 `with(obj) { }`가 호출 형태가 깔끔합니다.

## apply: this · 수신 객체 반환 {#fn-apply}

`apply`는 객체를 `this`로 받고, **객체 자신을 반환**합니다. 객체를 만든 직후 속성을 설정하고 그 객체를 그대로 돌려받는 빌더 패턴에 적합합니다.

```kotlin
inline fun <T> T.apply(block: T.() -> Unit): T { block(); return this }
```

```kotlin
val paint = Paint().apply {
    color = Color.RED      // this.color = ...
    isAntiAlias = true
    strokeWidth = 4f
}   // apply는 Paint 자신을 반환하므로 paint에 바로 할당된다
```

블록의 반환 타입이 `Unit`이라는 점에 주목해야 합니다. `apply`는 블록 결과를 버리고 **언제나 수신 객체를 반환**합니다.

## also: it · 수신 객체 반환 {#fn-also}

`also`는 객체를 `it`으로 받고, **객체 자신을 반환**합니다. 객체를 변형하지 않고 흘려보내면서 로깅·검증 같은 부수 효과만 끼워 넣을 때 씁니다.

```kotlin
inline fun <T> T.also(block: (T) -> Unit): T { block(this); return this }
```

```kotlin
val numbers = mutableListOf(1, 2, 3)
    .also { println("초기값: $it") }   // 부수 효과
    .also { it.add(4) }                 // it으로 원본 조작
// numbers == [1, 2, 3, 4]
```

`apply`와 `also`는 둘 다 객체를 반환하므로 체이닝에 쓰입니다. 차이는 객체를 `this`로 다루느냐(`apply`, 속성 설정에 유리) `it`으로 다루느냐(`also`, 객체를 인자로 넘기는 부수 작업에 유리)입니다.

## 수신 객체: this냐 it이냐 {#receiver}

스코프 함수의 동작을 좌우하는 핵심 개념이 **수신 객체(receiver)** 입니다. 함수 타입에는 두 형태가 있습니다.

- `(T) -> R` : 객체를 **일반 파라미터**로 받습니다. 람다 안에서 `it`(또는 명시한 이름)으로 접근합니다. → `let`, `also`
- `T.() -> R` : **수신 객체가 있는 함수 타입**입니다. 객체가 `this`가 되어, 멤버를 `this` 생략으로 부릅니다. → `run`, `with`, `apply`

`this`(수신 객체)와 `it`(인자)을 어떻게 고를지에는 실용적 기준이 있습니다.

- **객체의 멤버 프로퍼티·함수를 여러 번 만지면** `this` 계열(`run`/`apply`)이 코드가 짧습니다. `this.`를 생략할 수 있기 때문입니다.
- **객체를 다른 함수의 인자로 넘기거나, 이름을 부여해 가독성을 높이고 싶으면** `it` 계열(`let`/`also`)이 낫습니다. `it`을 `user ->`처럼 의미 있는 이름으로 바꿀 수 있습니다.
- **중첩될 때** `it` 계열이 안전합니다. `this`가 중첩되면 어느 수신 객체인지 모호해지지만, 바깥 `it`을 이름으로 바꾸면 충돌이 사라집니다.

```kotlin
outer.let { o ->
    inner.let { i ->
        // o, i로 두 수신 객체를 명확히 구분
    }
}
```

## 확장 함수의 원리 {#extension-mechanism}

확장 함수는 **클래스 외부에서 정의하지만 그 클래스의 멤버처럼 점(.) 호출로 부를 수 있는 함수**입니다. `fun 타입.함수명()` 형태로 선언하며, 이 점 앞의 타입을 **수신 타입(receiver type)**, 함수 안의 `this`를 **수신 객체**라고 부릅니다.

```kotlin
fun String.lastChar(): Char = this[this.length - 1]

"Kotlin".lastChar()   // 'n'
```

원리에서 정확히 알아야 할 점이 두 가지입니다.

**1) 확장 함수는 클래스에 실제로 추가되지 않습니다.** 컴파일러는 위 코드를 **수신 객체를 첫 인자로 받는 정적 메서드**로 바꿉니다. 자바로 보면 대략 다음과 같습니다.

```java
public static char lastChar(String $this) { ... }
// "Kotlin".lastChar()  →  StringExtKt.lastChar("Kotlin")
```

즉 확장 함수는 원본 클래스의 바이트코드를 건드리지 않습니다. 그래서 `String`처럼 우리가 소유하지 않은 클래스에도 함수를 "추가"할 수 있습니다.

**2) 확장 함수는 정적으로 디스패치됩니다(static dispatch).** 어떤 확장 함수가 호출될지는 **런타임 실제 타입이 아니라 컴파일 시점의 정적 타입**으로 결정됩니다. 멤버 함수의 가상 디스패치(오버라이드)와 다른 지점이라 면접에서 자주 나옵니다.

```kotlin
open class Animal
class Dog : Animal()

fun Animal.sound() = "동물 소리"
fun Dog.sound() = "멍멍"

val a: Animal = Dog()
println(a.sound())   // "동물 소리" — 변수의 정적 타입(Animal) 기준
```

`a`의 실제 객체는 `Dog`이지만, 변수의 선언 타입이 `Animal`이므로 `Animal.sound()`가 선택됩니다. 확장 함수는 다형성으로 오버라이드되지 않습니다.

여기서 따라 나오는 또 하나의 규칙은, **같은 시그니처의 멤버 함수가 있으면 항상 멤버가 우선**한다는 것입니다. 확장 함수는 멤버가 없을 때만 호출되므로, 확장 함수로 기존 멤버를 가릴 수 없습니다.

## 확장 프로퍼티 {#extension-property}

확장은 함수뿐 아니라 **프로퍼티**로도 정의할 수 있습니다. 문법은 멤버 프로퍼티와 비슷하지만, 본질적인 제약이 하나 있습니다.

**확장 프로퍼티는 backing field(뒷받침 필드)를 가질 수 없습니다.** 확장은 클래스에 실제 상태를 추가하지 못하기 때문입니다. 따라서 값을 저장할 수 없고, **반드시 getter(필요하면 setter)로 계산해서 제공**해야 합니다.

```kotlin
val String.lastChar: Char
    get() = this[length - 1]    // 매번 계산, 저장 불가

"Kotlin".lastChar   // 'n'
```

```kotlin
// 컴파일 에러: 확장 프로퍼티는 초기화(= 저장)를 할 수 없다
val String.cached: Int = 0   // Extension property cannot be initialized
```

확장 프로퍼티 역시 내부적으로는 **수신 객체를 인자로 받는 정적 getter 메서드**로 컴파일됩니다. 함수와 마찬가지로 정적 디스패치되며 상태를 추가하지 않습니다. 그래서 "저장이 아니라 파생 값"일 때만 확장 프로퍼티가 적절하고, 실제로 캐시·상태가 필요하면 확장 프로퍼티 대신 별도 설계가 필요합니다.

## 요약 {#summary}

> **TL;DR** — 스코프 함수 다섯 개는 "객체를 `this`로 보느냐 `it`으로 보느냐"와 "객체 자신을 반환하느냐 람다 결과를 반환하느냐" 두 축의 조합입니다. `let`(it·결과)·`run`(this·결과)·`with`(this·결과·비확장)·`apply`(this·객체)·`also`(it·객체). 이들의 토대인 수신 객체 개념은 확장 함수에서도 똑같이 쓰이며, 확장 함수·프로퍼티는 클래스를 수정하지 않고 수신 객체를 첫 인자로 받는 정적 멤버로 컴파일되어 **정적 디스패치**되고, 확장 프로퍼티는 backing field가 없어 계산형으로만 존재합니다.

1. **두 축**: 객체 노출(`this`/`it`)과 반환값(객체 자신/람다 결과)의 조합이 다섯 함수의 전부다.
2. **let·run·apply·also·with**: `let`=it·결과, `run`=this·결과, `with`=this·결과·비확장, `apply`=this·객체, `also`=it·객체.
3. **수신 객체**: `T.() -> R`는 객체를 `this`로, `(T) -> R`는 `it`으로 노출한다. 멤버를 많이 만지면 `this`, 인자로 넘기거나 이름이 필요하면 `it`.
4. **확장 함수 원리**: 클래스를 수정하지 않고 수신 객체를 첫 인자로 받는 정적 메서드로 컴파일되며, 정적 타입 기준으로 디스패치된다(오버라이드 안 됨, 멤버 우선).
5. **확장 프로퍼티**: backing field가 없어 값을 저장할 수 없고 getter로 계산만 제공한다.

<def title="Q) apply와 also는 둘 다 객체 자신을 반환하는데, 언제 무엇을 쓰나요?">

둘의 차이는 반환값이 아니라 객체를 람다 안에서 어떻게 노출하느냐입니다. `apply`는 객체를 `this`로 노출하므로 `color = ...`, `strokeWidth = ...`처럼 멤버 프로퍼티를 `this` 생략으로 설정하는 빌더 코드에 유리합니다. `also`는 객체를 `it`으로 노출하므로 `it`을 다른 함수에 인자로 넘기거나 로깅·검증 같은 부수 효과를 끼워 넣을 때 자연스럽습니다.

정리하면, 객체를 만든 직후 속성을 채워 그대로 돌려받는 설정 코드에는 `apply`, 객체를 변형하지 않고 흘려보내며 부수 작업만 추가하는 체이닝에는 `also`가 적합합니다. 둘 다 객체 자신을 반환하므로 체인 중간에 끼워도 흐름이 끊기지 않습니다.

</def>
<def title="Q) run과 let의 차이는 무엇이고, 각각 언제 적절한가요?">

반환 정책은 둘 다 같습니다 — 람다의 결과를 반환합니다. 차이는 객체 노출 방식입니다. `run`은 `block: T.() -> R`로 객체를 `this`로 받아 멤버를 `this` 생략으로 부르고, `let`은 `block: (T) -> R`로 객체를 `it`으로 받습니다.

따라서 객체의 멤버 프로퍼티·함수를 여러 번 호출해 하나의 값을 계산할 때는 `run`이 짧습니다. 반대로 nullable 객체를 `?.let { }`으로 null이 아닐 때만 처리하거나, 객체를 의미 있는 이름으로 바꿔 다른 함수에 넘길 때는 `let`이 낫습니다. 중첩 상황에서도 `it`을 이름으로 바꿀 수 있는 `let`이 수신 객체 모호성을 피하기 좋습니다.

</def>
<def title="Q) 확장 함수는 다형성(오버라이드)으로 동작하나요?">

아니요. 확장 함수는 정적으로 디스패치됩니다. 어떤 확장 함수가 호출될지는 런타임의 실제 객체 타입이 아니라 컴파일 시점의 정적 타입(변수의 선언 타입)으로 결정됩니다.

예를 들어 `Animal.sound()`와 `Dog.sound()`를 각각 확장 함수로 정의하고 `val a: Animal = Dog()`에서 `a.sound()`를 호출하면, 실제 객체가 `Dog`이라도 변수 타입이 `Animal`이므로 `Animal.sound()`가 선택됩니다. 이는 확장 함수가 클래스에 추가되는 게 아니라 수신 객체를 첫 인자로 받는 정적 메서드로 컴파일되기 때문입니다. 또한 같은 시그니처의 멤버 함수가 있으면 항상 멤버가 우선하므로, 확장 함수로 기존 멤버를 가릴 수도 없습니다.

</def>
<def title="Q) 확장 프로퍼티에 값을 저장할 수 있나요? 왜 그런가요?">

저장할 수 없습니다. 확장 프로퍼티는 backing field(뒷받침 필드)를 가질 수 없기 때문입니다. 확장은 원본 클래스의 바이트코드를 수정하지 않고 외부에서 정적 멤버로 컴파일되는 것이라, 클래스 인스턴스에 새로운 상태(필드)를 추가할 방법이 없습니다.

그래서 확장 프로퍼티는 `= 0` 같은 초기화를 할 수 없고, 반드시 getter로 매번 계산해서 값을 제공해야 합니다(필요하면 var로 setter도 둘 수 있지만 역시 다른 저장소에 위임해야 합니다). 결론적으로 확장 프로퍼티는 저장된 상태가 아니라 수신 객체로부터 파생되는 계산 값에만 적합하며, 실제 캐싱이나 상태가 필요하면 확장이 아닌 다른 설계를 써야 합니다.

</def>
<def title="Q) ?.let과 스마트캐스트는 같은 것인가요?">

다릅니다. `name?.let { }`에서 블록이 null일 때 실행되지 않는 이유는 스마트캐스트가 아니라 안전 호출 연산자 `?.` 때문입니다. `name`이 null이면 `?.`가 `let` 호출 자체를 건너뛰고 전체 식이 null로 평가되며, null이 아니면 그 값이 `let`의 수신 객체로 들어가 람다 파라미터 `it`은 non-null 타입이 됩니다.

스마트캐스트는 `if (name != null) { name.length }`처럼 컴파일러가 null 체크 이후 변수의 타입을 non-null로 좁혀 주는 별개의 기능입니다. 특히 가변 프로퍼티나 다른 스레드가 바꿀 수 있는 값은 스마트캐스트가 안 되는 경우가 있는데, 이때 `?.let`은 값을 람다 파라미터로 고정해 받으므로 안전하게 non-null로 다룰 수 있다는 실용적 차이가 있습니다.

</def>
