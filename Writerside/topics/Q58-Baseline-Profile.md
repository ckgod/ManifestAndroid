# Q58) Baseline Profile

## Baseline Profile은 무엇인가요? {#what-is-baseline-profile}

Baseline Profile은 앱의 시작 시간과 런타임 실행 성능을 끌어올리기 위해 제공되는 Android의 성능 최적화 기능입니다. 미리 컴파일된 코드 정보를 함께 배포함으로써 인터프리터 단계와 <tooltip term="JIT">JIT</tooltip> 컴파일 단계를 우회하여 더 빠른 실행을 가능하게 합니다. 첫 실행부터 대략 20–30%의 속도 향상이 기대되며, 더 매끄럽고 효율적인 사용자 경험을 만드는 데 기여합니다. Android Runtime(ART)은 이 프로필을 활용해 앱 설치 시점에 핵심 코드 경로를 미리 컴파일함으로써 반응성을 높이고 시작 지연을 줄입니다.

Baseline Profile은 프로필에 명시된 코드 경로에 대해 <tooltip term="AOT">AOT</tooltip> 컴파일을 수행함으로써 동작합니다. 프로필에는 클래스와 메서드 정보가 포함되어 있고, ART는 설치 단계에서 이 정보를 토대로 해당 부분을 사전 컴파일합니다. 라이브러리 작성자 입장에서도 Baseline Profile을 함께 배포하면, 그 라이브러리를 가져다 쓰는 앱이 별다른 수고 없이 성능 최적화를 그대로 누릴 수 있다는 장점이 있습니다.

### Baseline Profile의 동작 방식 {#how-it-works}

1. **핵심 코드 경로 정의**: 개발자는 앱이 시작되는 순간부터 사용자가 가장 자주 거치는 흐름을 프로파일링하여, 성능에 결정적인 메서드와 클래스를 식별합니다.
2. **프로필 생성**: Jetpack Macrobenchmark 라이브러리 같은 도구를 사용해 앱 동작을 측정·기록함으로써 중요한 코드 경로를 추출하고 프로필을 생성합니다.
3. **프로필 포함**: 생성된 Baseline Profile은 APK 또는 AAB에 함께 번들링되어 사용자에게 배포됩니다.
4. **설치 시 최적화**: 사용자가 기기에 앱을 설치하면, ART는 프로필에 명시된 메서드와 클래스를 네이티브 코드로 미리 컴파일하여 둡니다.

AGP 8.0 이상부터는 Baseline Profile Gradle 플러그인을 사용할 수 있습니다. 이 플러그인은 Baseline Profile 생성 과정을 단순화하고, 패키지 필터링 옵션과 flavor 제어 같은 편의 기능을 제공합니다. 가이드에 따라 Baseline Profile을 생성하면 각 모듈의 `/src/main/generated/baselineProfiles` 디렉터리에 `baseline-prof.txt`라는 이름의 텍스트 파일이 만들어지며, 그 안에는 다음과 같이 클래스와 메서드 선언이 나열되어 있습니다.

Google이 직접 개발·운영하는 [Now in Android](https://github.com/android/nowinandroid) 저장소에서 실제 구현 예시를 확인할 수 있습니다. 이 저장소에는 화면별 Baseline Profile 생성 방법, 다양한 사용자 동선 시나리오, 그리고 성능을 효과적으로 최적화하는 구체적인 구성 방식이 정리되어 있습니다.

### 요약 {#summary}

<tldr>
Baseline Profile은 핵심 코드 경로를 미리 컴파일해 앱 시작 시간을 줄이고 런타임 실행을 매끄럽게 만들어 주는 성능 최적화 도구입니다. Jetpack Macrobenchmark 같은 도구로 핵심 경로를 식별·정의하면, 다양한 디바이스에서 더 빠르고 반응성 높은 사용자 경험을 일관되게 제공할 수 있습니다. 더 깊이 있는 활용법은 *Improve Your Android App Performance With Baseline Profiles* 같은 자료를 참고하면 좋습니다.
</tldr>

<def title="Q) Android Runtime(ART)은 Baseline Profile을 어떻게 활용해 앱 성능을 끌어올리며, 전통적인 JIT 컴파일과 비교했을 때 어떤 핵심 장점이 있나요?">

ART는 앱 설치 시점에 APK/AAB에 포함된 Baseline Profile을 읽어, 그 안에 나열된 클래스와 메서드를 AOT 방식으로 미리 네이티브 코드로 컴파일해 둡니다. 그 결과 첫 실행부터 핵심 경로의 메서드들이 인터프리터를 거치거나 JIT가 임계치를 채울 때까지 기다릴 필요 없이 곧바로 컴파일된 코드로 실행됩니다. 본문에서 언급된 20–30%대의 시작 속도 향상은 바로 이 "첫 사용 시점부터 사전 컴파일된 코드를 사용한다"는 점에서 얻어지는 효과입니다.

전통적인 JIT 컴파일은 런타임 동안 메서드 호출 패턴을 모니터링하다가, 특정 임계치를 넘긴 핫스팟만 점진적으로 컴파일해 가는 방식입니다. 이 방식은 실제 사용 데이터에 기반해 최적화한다는 장점이 있지만, 그 결과를 누리려면 결국 "충분한 워밍업 시간"이 필요하고, 워밍업 동안에는 인터프리터 실행과 JIT 컴파일 자체의 비용이 동시에 발생합니다. 이 워밍업 비용은 사용자가 가장 민감하게 느끼는 앱 시작 직후와 첫 화면 진입 시점에 가장 크게 누적되며, 결과적으로 초기 사용 경험을 떨어뜨립니다.

Baseline Profile은 이 간극을 메워 줍니다. JIT처럼 사용 데이터를 활용해 어떤 코드가 핵심인지 골라내되, 그 결과를 빌드 산출물의 일부로 박제해 사용자 디바이스로 미리 가져갑니다. 따라서 첫 실행에서도 워밍업 없이 핵심 경로가 빠르게 동작하고, 디바이스 간 성능 편차도 줄어듭니다. JIT는 여전히 프로필에 잡히지 않은 코드 경로를 보완해 주는 역할로 함께 동작하기 때문에, 두 방식은 대체 관계라기보다는 상호 보완 관계로 보는 것이 정확합니다.
