# Q71) HTTP와 상태 코드

## HTTP 요청·응답 구조 {#http-message}

HTTP(HyperText Transfer Protocol)는 클라이언트가 **요청(request)**을 보내고 서버가 **응답(response)**을 돌려주는, 요청-응답 기반의 무상태(stateless) 프로토콜입니다. 무상태란 서버가 각 요청을 독립적으로 처리하고 이전 요청의 맥락을 기억하지 않는다는 뜻입니다. 그래서 로그인 상태 같은 맥락은 쿠키, 토큰 등으로 매 요청에 직접 실어 보내야 합니다.

요청과 응답 메시지는 모두 다음 세 부분으로 구성됩니다.

### 요청 메시지 {#request-structure}

```http
POST /api/users HTTP/1.1        ← 시작 줄: 메서드 + 경로 + 버전
Host: api.example.com           ← 헤더: 메타데이터(키-값)
Content-Type: application/json
Authorization: Bearer eyJ...
                                ← 빈 줄(헤더와 본문의 경계)
{"name": "kim", "age": 30}      ← 본문(body): 전송할 데이터
```

- **시작 줄(start line)**: 메서드, 요청 대상 경로, 프로토콜 버전을 담습니다.
- **헤더(headers)**: `Host`, `Content-Type`, `Authorization` 등 요청에 대한 메타데이터입니다.
- **본문(body)**: 실제로 전송하는 데이터입니다. `GET`처럼 본문이 없는 요청도 많습니다.

### 응답 메시지 {#response-structure}

```http
HTTP/1.1 200 OK                 ← 상태 줄: 버전 + 상태 코드 + 사유 구문
Content-Type: application/json
Content-Length: 27
                                ← 빈 줄
{"id": 1, "name": "kim"}        ← 본문
```

응답의 시작 줄은 **상태 줄(status line)**이라 부르며, 이 줄에 들어가는 **상태 코드(status code)**가 이 토픽의 핵심입니다. 클라이언트는 본문을 파싱하기 전에 상태 코드만 보고도 요청의 성공·실패와 그 분류를 즉시 알 수 있습니다.

## HTTP 메서드 {#http-methods}

메서드는 **자원(resource)에 대해 무엇을 하려는지**를 나타내는 동사입니다. 같은 URL이라도 메서드에 따라 의미가 완전히 달라집니다.

| 메서드 | 의미 | 본문 | 안전(safe) | 멱등(idempotent) |
|--------|------|------|-----------|------------------|
| GET | 자원 조회 | 없음 | O | O |
| POST | 자원 생성·처리 요청 | 있음 | X | X |
| PUT | 자원 전체 교체(없으면 생성) | 있음 | X | O |
| PATCH | 자원 부분 수정 | 있음 | X | X |
| DELETE | 자원 삭제 | 보통 없음 | X | O |
| HEAD | GET과 같되 본문 없이 헤더만 | 없음 | O | O |

- **안전(safe)**: 서버의 상태를 변경하지 않는 읽기 전용 메서드입니다. `GET`, `HEAD`가 해당합니다.
- **멱등(idempotent)**: 같은 요청을 여러 번 보내도 결과 상태가 한 번 보낸 것과 같은 성질입니다. 다음 절에서 자세히 다룹니다.

`PUT`과 `POST`의 구분이 면접에서 자주 나옵니다. `PUT`은 **클라이언트가 자원의 위치(URI)를 지정**하고 그 위치를 통째로 교체합니다. `POST`는 **서버가 위치를 정하는 새 자원 생성**이나, CRUD로 깔끔히 나뉘지 않는 처리 요청(결제, 검색 등)에 씁니다.

## 2xx · 4xx · 5xx의 의미 {#status-classes}

상태 코드는 세 자리 숫자이고, 첫 자리가 다섯 개의 분류를 나타냅니다.

| 분류 | 의미 | 책임 소재 |
|------|------|----------|
| 1xx | 정보(처리 진행 중) | — |
| 2xx | 성공 | — |
| 3xx | 리다이렉션(추가 동작 필요) | 클라이언트가 후속 요청 |
| 4xx | 클라이언트 오류 | **요청을 보낸 쪽** |
| 5xx | 서버 오류 | **응답하는 서버 쪽** |

4xx와 5xx의 구분이 가장 중요합니다. **4xx는 "네 요청이 잘못됐다", 5xx는 "내(서버) 쪽에서 문제가 났다"**를 뜻합니다. 이 구분이 클라이언트의 재시도 전략을 결정합니다. 4xx는 같은 요청을 다시 보내도 결과가 같으므로 요청 자체를 고쳐야 하고, 5xx는 일시적 장애일 수 있어 재시도가 유효할 수 있습니다.

### 2xx 성공 {#success-2xx}

- **200 OK**: 요청 성공. `GET` 응답에 가장 흔합니다.
- **201 Created**: 새 자원이 생성됨. `POST`로 자원을 만들었을 때 적절하며, 보통 `Location` 헤더에 새 자원의 URI를 담습니다.
- **204 No Content**: 성공했지만 응답 본문이 없음. `DELETE`나 본문이 필요 없는 `PUT` 응답에 씁니다.

### 4xx 클라이언트 오류 {#client-error-4xx}

- **400 Bad Request**: 요청 형식·문법 자체가 잘못됨(깨진 JSON 등).
- **401 Unauthorized**: 인증되지 않음. 이름과 달리 "인증(authentication) 실패"입니다. 토큰이 없거나 만료됐을 때입니다.
- **403 Forbidden**: 인증은 됐으나 **권한(authorization)이 없음**. 누구인지는 알지만 그 행위를 할 자격이 없는 경우입니다.
- **404 Not Found**: 요청한 자원이 존재하지 않음.
- **409 Conflict**: 현재 자원 상태와 요청이 충돌함(중복 가입 등).
- **429 Too Many Requests**: 요청 빈도 제한 초과. `Retry-After` 헤더로 재시도 시점을 알려 줄 수 있습니다.

401과 403의 구분은 단골 질문입니다. **401은 "네가 누구인지 모르겠다"(인증), 403은 "누구인지는 알지만 안 된다"(인가)**입니다.

### 5xx 서버 오류 {#server-error-5xx}

- **500 Internal Server Error**: 서버 내부에서 처리되지 않은 예외가 발생함. 가장 포괄적인 서버 오류입니다.
- **502 Bad Gateway**: 게이트웨이·프록시가 상위 서버로부터 잘못된 응답을 받음.
- **503 Service Unavailable**: 서버가 일시적으로 처리 불가(과부하·점검). 보통 일시적이라 재시도 대상입니다.
- **504 Gateway Timeout**: 게이트웨이가 상위 서버 응답을 제한 시간 안에 받지 못함.

## 멱등성 {#idempotency}

### 정의 {#idempotency-definition}

멱등성(idempotency)이란 **같은 요청을 한 번 보내든 여러 번 보내든 서버의 최종 상태가 동일한** 성질입니다. 응답 본문이나 상태 코드가 매번 같아야 한다는 뜻이 **아니라**, 요청이 만들어 내는 **부수 효과(서버 상태 변화)**가 같다는 뜻입니다.

### 메서드별 멱등성 {#idempotency-by-method}

- **GET / HEAD**: 멱등. 읽기만 하므로 상태를 바꾸지 않습니다(안전하므로 자명하게 멱등).
- **PUT**: 멱등. `PUT /users/1`로 같은 데이터를 100번 보내도 자원 1번은 그 데이터로 한 번 교체된 상태와 동일합니다.
- **DELETE**: 멱등. `DELETE /users/1`을 두 번 보내면 두 번째는 404가 올 수 있지만, **"자원 1번이 없다"는 최종 상태는 동일**합니다. 상태 코드가 달라도 멱등성은 유지됩니다.
- **POST**: **비멱등**. `POST /users`를 두 번 보내면 보통 사용자가 둘 생성됩니다. 호출 횟수만큼 상태가 누적됩니다.
- **PATCH**: 일반적으로 **비멱등**. 단, 절대값으로 덮어쓰는 PATCH는 멱등일 수 있어 구현에 달려 있습니다. 예를 들어 `{"count": 5}`로 덮어쓰면 멱등이지만, `{"op": "increment"}`처럼 상대적 변경이면 비멱등입니다.

### 왜 중요한가 {#idempotency-why}

멱등성은 **재시도의 안전성**을 결정합니다. 네트워크가 불안정한 모바일 환경에서는, 요청은 서버에 도달했는데 응답이 유실되어 클라이언트가 실패로 오인하는 일이 흔합니다. 이때 멱등한 요청은 그냥 다시 보내면 됩니다. 두 번 도달해도 상태가 망가지지 않기 때문입니다.

반면 비멱등한 `POST`를 무턱대고 재시도하면 결제가 두 번 일어나는 식의 중복이 생깁니다. 그래서 결제처럼 위험한 `POST`에는 클라이언트가 고유한 **멱등성 키(idempotency key)**를 헤더에 실어 보내고, 서버가 같은 키의 중복 요청을 걸러 내 멱등성을 인위적으로 부여하는 패턴을 씁니다.

```kotlin
// OkHttp Interceptor: 안전하게 재시도할 수 있는 멱등 메서드만 자동 재시도
class RetryInterceptor(private val maxRetry: Int = 2) : Interceptor {
    private val idempotent = setOf("GET", "HEAD", "PUT", "DELETE")

    override fun intercept(chain: Interceptor.Chain): Response {
        val request = chain.request()
        var response = chain.proceed(request)
        var tries = 0
        // 5xx(서버 오류)이고 멱등 메서드일 때만 재시도한다
        while (response.code in 500..599 &&
            request.method in idempotent &&
            tries < maxRetry
        ) {
            response.close()
            tries++
            response = chain.proceed(request)
        }
        return response
    }
}
```

위 코드의 두 조건이 핵심입니다. **5xx(서버 측 일시 오류)일 때만** 재시도하고(4xx는 재시도해도 똑같이 실패), **멱등 메서드일 때만** 재시도합니다(POST 재시도는 중복 생성 위험).

## 요약 {#summary}

> **TL;DR** — HTTP는 무상태 요청-응답 프로토콜로, 메시지는 시작 줄·헤더·본문으로 구성됩니다. 메서드는 자원에 대한 동작(GET/POST/PUT/PATCH/DELETE)을, 상태 코드의 첫 자리는 분류(2xx 성공 / 4xx 클라이언트 오류 / 5xx 서버 오류)를 나타냅니다. 멱등성은 같은 요청을 여러 번 보내도 서버 상태가 같은 성질이며, 재시도 안전성을 결정합니다 — 5xx + 멱등 메서드만 재시도하는 것이 안전한 기본 전략입니다.

1. **요청·응답 구조**: 두 메시지 모두 시작 줄(요청은 메서드/경로, 응답은 상태 코드) + 헤더 + 빈 줄 + 본문으로 구성된다. HTTP는 무상태라 맥락은 매 요청에 직접 실어야 한다.
2. **메서드**: 자원에 대한 동작을 나타내는 동사. 안전(GET/HEAD)·멱등(GET/HEAD/PUT/DELETE) 속성으로 분류된다. PUT은 위치 지정 교체, POST는 생성·처리.
3. **2xx·4xx·5xx 의미**: 첫 자리가 분류다. 2xx 성공, 4xx는 요청한 쪽 잘못, 5xx는 서버 쪽 문제. 이 구분이 재시도 전략을 가른다.
4. **멱등성**: 같은 요청을 N번 보내도 서버 최종 상태가 1번과 같은 성질. GET/PUT/DELETE는 멱등, POST는 비멱등. 재시도 안전성의 기준이 된다.

<def title="Q) HTTP 요청과 응답 메시지는 어떤 구조로 되어 있나요?">

요청과 응답 모두 세 부분으로 구성됩니다. 첫째는 시작 줄입니다. 요청에서는 메서드, 요청 경로, 프로토콜 버전이 들어가고, 응답에서는 상태 줄이라 부르며 버전, 상태 코드, 사유 구문이 들어갑니다. 둘째는 헤더로, `Host`, `Content-Type`, `Authorization` 같은 키-값 메타데이터입니다. 셋째는 본문으로 실제 전송 데이터이며, 헤더와 본문은 빈 줄로 구분됩니다.

HTTP는 무상태 프로토콜이라 서버가 각 요청을 독립적으로 처리하고 이전 맥락을 기억하지 않습니다. 그래서 로그인 같은 상태는 쿠키나 토큰으로 매 요청에 직접 실어 보내야 합니다.

<def title="Q) 401과 403의 차이는 무엇인가요?">

둘 다 4xx 클라이언트 오류지만 의미가 다릅니다. 401 Unauthorized는 인증(authentication) 실패로, 서버가 요청한 주체가 누구인지 확인하지 못한 상태입니다. 토큰이 없거나 만료됐을 때 발생하며, 올바른 자격 증명으로 다시 시도하면 통과할 수 있습니다.

403 Forbidden은 인가(authorization) 실패로, 주체가 누구인지는 확인됐지만 그 행위를 할 권한이 없는 상태입니다. 다른 자격 증명을 줘도 같은 권한이라면 계속 거부됩니다. 한 줄로 정리하면 401은 "네가 누구인지 모르겠다", 403은 "누구인지는 알지만 안 된다"입니다.

</def>
<def title="Q) 4xx와 5xx를 구분하는 것이 클라이언트 입장에서 왜 중요한가요?">

책임 소재와 재시도 전략이 다르기 때문입니다. 4xx는 요청을 보낸 클라이언트 쪽 잘못입니다. 잘못된 형식, 없는 자원, 권한 부족 등이며, 같은 요청을 다시 보내도 결과가 똑같으므로 요청 자체를 고쳐야 합니다. 무작정 재시도하면 의미 없이 실패만 반복합니다.

5xx는 응답하는 서버 쪽 문제입니다. 특히 503 Service Unavailable이나 504 Gateway Timeout은 과부하나 일시적 장애일 수 있어, 잠시 후 같은 요청을 다시 보내면 성공할 수 있습니다. 그래서 5xx는 백오프를 두고 재시도하는 것이 합리적입니다. 단, 재시도는 멱등한 메서드에 한해 안전합니다.

</def>
<def title="Q) 멱등성이 무엇이고, 어떤 메서드가 멱등한가요?">

멱등성은 같은 요청을 한 번 보내든 여러 번 보내든 서버의 최종 상태가 동일한 성질입니다. 응답 코드나 본문이 매번 같아야 한다는 뜻이 아니라, 요청이 만드는 서버 상태 변화가 같다는 뜻입니다.

GET과 HEAD는 읽기만 하므로 멱등합니다. PUT은 같은 데이터로 자원을 통째로 교체하므로 몇 번 보내도 결과 상태가 같아 멱등합니다. DELETE도 멱등한데, 두 번째 호출에서 404가 오더라도 "자원이 없다"는 최종 상태는 동일하기 때문입니다. 상태 코드가 달라도 멱등성은 유지됩니다. 반면 POST는 호출할 때마다 새 자원이 누적 생성되므로 비멱등이고, PATCH는 구현에 따라 다릅니다.

</def>
<def title="Q) 네트워크 실패 시 요청을 재시도할 때 무엇을 주의해야 하나요?">

두 가지를 봐야 합니다. 첫째는 상태 코드입니다. 4xx는 요청 자체가 잘못된 것이라 재시도해도 같은 실패가 반복되므로 재시도 대상이 아닙니다. 5xx(특히 503, 504)는 서버의 일시적 장애일 수 있어 백오프 후 재시도가 유효합니다.

둘째는 메서드의 멱등성입니다. 멱등한 GET/PUT/DELETE는 중복 도달해도 상태가 망가지지 않아 안전하게 재시도할 수 있습니다. 비멱등한 POST는 응답만 유실되고 요청은 도달한 경우 재시도가 중복 생성(예: 이중 결제)을 일으킬 수 있어 위험합니다. 위험한 POST를 재시도해야 한다면, 클라이언트가 고유한 멱등성 키를 헤더에 실어 보내고 서버가 같은 키의 중복 요청을 걸러 내 멱등성을 인위적으로 부여하는 패턴을 씁니다.
